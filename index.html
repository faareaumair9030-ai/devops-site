01 — what even is devops

the wall between devs and ops

devops is basically just the idea that your dev team and ops team shouldnt be two seperate things fighting each other lol. like back in the day devs would write code, throw it over the wall to ops, and ops would be like "what is this mess" and everyone was miserable.

devops says nah — lets just make it one workflow. same team, same goals, same pipeline. its more of a culture shift than a tool or a job title, even tho "devops engineer" is definitely a job title now.

the goal isnt to make devs do ops work. its to remove the friction between the two so things actually ship without drama.
02 — continuous integration

CI — catch the bug
before it bites you

CI stands for continuous integration. basically every time someone pushes code, u run all your tests automatically so u catch broken stuff early before it becomes a huge problem.

nobody wants to find out 3 weeks later that someone commited a bug that broke half the app. CI makes sure that the moment bad code lands — the pipeline screams and everyone knows.

$ git push origin feature/new-auth ▶ Triggering CI pipeline... ✔ lint passed (2s) ✔ unit tests passed (14s) ✔ integration tests passed (38s) ✔ build successful ✔ PR is ready to merge

thats the dream. small commits, fast feedback, no suprises at 2am on a friday before a release.

03 — continuous delivery vs deployment

CD — delivery or
deployment?

CD is continuous delivery or continuous deployment depending on who u ask. and yes its confusing that they're both CD.

delivery means the code is always in a state where it could go to production. like its packaged, tested, ready — but a human still presses the button to deploy.

deployment means it actually does go to production automatically. no human, no button. code merges, pipeline runs, users get the update.

a lot of teams do delivery first and then add full deployment later once they trust the pipeline enough. makes sense tbh.

05 — the actual point

ship smaller things,
more often

the whole point is to ship smaller changes more often instead of big scary releases every few months.

smaller changes = smaller blast radius when something goes wrong. and stuff does go wrong — pipelines fail, tests catch regressions, deployments roll back.

thats kind of the point tho. fail fast and fix fast rather then letting problems pile up into one catastrophic release night.

$ git push origin main ▶ Running pipeline... ✘ test failed: auth_token_expiry_spec.rb ▶ Rolling back deploy. Notifying team. ✔ Production stable. Fix it and push again.

honestly once u have a decent CI/CD setup running its hard to imagine going back. just push code and let the pipeline figure out if its good or not.

devops isnt a role, its a mindset. CI/CD isnt magic, its just automation doing the boring stuff so ur team can focus on actually building things. set it up once, tweak it forever, thank yourself later.
